# Research note — generated/typed Telegram bot clients in Go and Rust

**Date:** 2026-06-15
**Status:** High-level summary from public documentation, **not a code audit.** Every claim below is sourced from each project's public README / docs as of this date; specifics beyond these well-known facts are intentionally omitted rather than guessed.

This note backs the precedents cited in `ARCHITECTURE.md` ADR-001 ("generated concrete methods") and ADR-010 ("JSON request body when no file" — the alternative weighed and rejected). It exists so those precedent arguments are verifiable rather than dangling references.

---

## gotgbot (Go) — precedent for ADR-001

- **Repo / docs:** <https://github.com/PaulSonOfLars/gotgbot> · <https://pkg.go.dev/github.com/PaulSonOfLars/gotgbot/v2>

gotgbot is an **autogenerated** Go wrapper for the Telegram Bot API: all Telegram types and methods are code-generated from a machine-readable Bot API spec into `gen_*.go` files, so the surface is "guaranteed to match the docs" and updating is mechanical. Regeneration is run via `go generate` from the repo root against the spec commit pinned in a `spec_commit` file (with an upgrade switch to bump to the latest spec). The generated API is **type-safe with no `interface{}` logic** — every type mirrors the documented Bot API type — and the library depends only on the Go **standard library** (no third-party HTTP/JSON bloat).

**How it informs ADR-001.** This is the direct precedent for ADR-001's decision to generate **concrete methods** onto a single client rather than route everything through a reflective/dynamic dispatcher: a real generated method per Bot API method gives docs-faithful types, mechanical updates, and ordinary debuggability — exactly the trade gotgbot has shipped in Go. v2 mirrors the "regenerate from spec, no hand-written bodies" discipline (our `scripts/api-parser.ts` plays the role of gotgbot's `go generate`).

## teloxide (Rust) — precedent for ADR-010

- **Repo / docs:** <https://github.com/teloxide/teloxide> · <https://docs.rs/teloxide/>

teloxide is a full-featured Rust bot framework. It uses **`serde`** for (de)serialization of Telegram types. API calls are modeled as **typed request payload structs** — for example `teloxide::payloads::SetWebhook` — and are issued through the `Requester` trait implemented for `Bot` (so a call like `bot.send_message(chat_id, text)` builds and sends the corresponding typed payload). It supports both **long polling and webhooks** (the latter via its `dispatching::update_listeners::webhooks` module). Requests are sent with a **JSON request body** (serde-serialized payloads), with multipart used where file uploads are involved.

**How it informs ADR-010.** teloxide is the precedent for the alternative ADR-010 *considered and rejected*: sending an `application/json` body for file-less calls, which would let structured fields remain nested objects serialized in one pass rather than per-field strings. ADR-010 weighs this fairly — it is a proven design — but chooses uniform form encoding instead, because v2's ADR-002 already makes structured fields `Json<T>` strings produced at the call site, so a single all-strings field model needs nothing the JSON body would add, while avoiding a second body model. teloxide shows the JSON-body path is viable; ADR-010 explains why it buys v2 nothing extra.
